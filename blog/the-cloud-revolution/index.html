The Cloud Revolution

December 6, 2016

Server-side development is rapidly moving into the world of declarative and functional programming. Even Java 8 now supports functional programming constructs such as lambda syntax and a functional-style collections API. Libraries such as Javaslang are bringing even more powerful functional programming to Java developers.

/* filtering by age and converting to people to names in Java 8 with Javaslang */
List<String> names = persons
    .filter(p -> p.age > 12)
    .map(Person::getName);

This shift towards functional programming is not accidental.

In 2006, Ruby on Rails gave PHP developers a reasonable alternative to web development. Ruby on Rails in particular inspired a whole legion of programmers to rethink standards like the JCP and build sane frameworks rather than work within the confines of slow-moving standards designed by committee.

While this was happening, Amazon was quietly revolutionizing the data center.

Aside from Ruby on Rails, most frameworks were initially slow to catch onto the revolution happening in the cloud. Services like Heroku and AWS began to bring unlimited resources on-tap to developers. For awhile it even looked like the JVM would lose relevance with new companies like Twitter embracing Ruby on Rails rather than the ever-complex bloatware dictated by ivory tower standards committees.

While on-tap resources brought tremendous flexibility to developers, they also introduced a host of situations that popular frameworks of the day were not equipped to deal with.

The Cloud

Let’s be clear; the cloud is just a fancy phrase that translates to ”other people’s computers”. When you entrust your applications to computers that you don’t own, expect the unexpected; the main issue affecting cloud architectures is that a machine in the cloud can disappear at any time and for any reason. This is the core reason why monolithic-style frameworks that hold state on the server ill-equipped as a cloud-based technology. As much as the big players like IBM and Oracle want to reposition their container-based technologies as cloud-ready, it will never happen. The reality is that a long-time ago, while Ruby on Rails was quietly embracing HTTP at the core of its offering, the Java world was attempting to graft the object model on top of HTTP and essentially hide the web from developers. Bad move.

Fast forward to today, and now the glue between servers — the network — has in many ways become more robust than servers themselves. This pushes resilience concerns off of servers and onto the glue that connects servers together. If we consider the network as the pipes, we also need a fabric that sits on top which adds context to those connections. Rather than containers, we have a fabric - what we refer to as a cluster - which needs to be managed across potentially hundreds of servers.

Resilience in the Cloud

If we don’t expect our servers to stay alive, the very least we require is that if they die they spin back up in a consistent state. This is trivial for stateless applications, but non-trivial for stateful applications.
